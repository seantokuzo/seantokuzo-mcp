## 0. Executive summary

Phase 2.5e closes the last open threat vector in Kuzo MCP's 5-layer security model (see `docs/SECURITY.md`): **compromised plugin packages**. With 2.5a–d in place (manifest hardening, credential broker, consent + audit, process isolation), we now make plugins installable, verifiable, and updateable without ever trusting unsigned code or auto-executing updates.

The four load-bearing pieces:

1. **Part A (monorepo):** Split the flat `src/` tree into `packages/{types,core,cli,plugin-github,plugin-jira,plugin-git-context}` under pnpm workspaces so each package publishes independently to npm.
2. **Part B (release):** GitHub Actions release workflow with Changesets + npm Trusted Publishing (OIDC, tokenless) + provenance attestations, tied to first-party trust (`github.com/seantokuzo/*`).
3. **Part C (verify):** `kuzo plugins install` fetches the npm provenance attestation from the registry, verifies the Sigstore signature + SLSA claims, and rejects any package that fails. No `npm install` runs before verification passes.
4. **Part D (CLI):** `kuzo plugins install/update/rollback/list/uninstall` with atomic install, isolated per-plugin `node_modules`, retained versions for rollback, consent re-run on capability changes, and never auto-update.

### Locked decisions (from prior research cycle)

| # | Decision | Rationale |
|---|---|---|
| 1 | **pnpm workspaces only**, no Turborepo (yet) | Archetype proven by `modelcontextprotocol/typescript-sdk`. Turbo earns its keep at 20+ packages or remote cache needs. ~30 min to add later if pain emerges. |
| 2 | **Scoped `@kuzo-mcp/*` names** | Enables oclif-style friendly-name resolution (`install github` → `@kuzo-mcp/plugin-github`), clean blessed-vs-community distinction via scope. |
| 3 | **Option A verification** — pre-install attestation check via npm registry API + Sigstore, gate install scripts until verified | Matches the "never run untrusted code" principle. Programmatic; we own the failure mode. (See Part C.) |
| 4 | **Exact-name MVP install** — no search/discovery | Deferred. `kuzo plugins install github` works; `kuzo plugins search` is a post-2.5e follow-up. |
| 5 | **Tokenless day-one** via Trusted Publishing (OIDC, no `NPM_TOKEN`) | GA since July 2025; nothing to gain by delaying. |
| 6 | **Rollback semantics: keep last 3 versions per plugin**, manual rollback to previous or any retained version | Matches oclif/homebrew intuition; re-runs consent on rollback just like install. |

### Non-goals (explicitly out of 2.5e)

- **Plugin registry / discovery** — no `kuzo plugins search`, no curated directory. MVP requires exact package name. (Defer.)
- **Auto-update** — never. Intentional friction, citing ShadyPanda Dec 2025 auto-update incident.
- **Private registries / non-npm sources** — out. `--registry` override is accepted; Git URLs, `file:` specs, and tarball installs are refused in v1.
- **Plugin hot-reload** — out. Restart required after update.
- **Versioning codemods** — out. Changesets note-files by hand; generators land post-2.5e.
- **Org-level Trusted Publisher** — npm doesn't offer this (Apr 2026). We configure per-package.

### Cross-cutting build order

The four parts have dependencies — this is the recommended order. Each step is a commit.

1. **A: Monorepo skeleton** (Part A §1, steps 1-7) — restructure to `packages/*`, delete legacy `src/`. Repo green at every step.
2. **A: Project references + parity test** (Part A §1, steps 8-10) — incremental builds, dev-to-install parity gate.
3. **B: Changesets + per-package publish fields** (Part B §2-4) — version/changelog machinery, publishConfig, repository.directory on every package.
4. **B: npmjs.com setup** (Part B §5) — reserve scope, bootstrap 6 packages with placeholder publish, configure Trusted Publisher, revoke bootstrap token.
5. **B: release.yml** (Part B §1) — the workflow itself; first real release goes via `@kuzo-mcp/types` canary.
6. **C+D: Install CLI core** (Part C full, Part D §2) — attestation fetch, Sigstore verify, trust policy, atomic install, consent integration.
7. **D: Update + rollback** (Part D §3-4) — version retention, capability diff, manual update gate.
8. **E: Docs + acceptance** (Part E) — update `SECURITY.md`, `PLANNING.md`, `STATE.md`. Close phase.

## Part A — Monorepo restructure

> Split Kuzo MCP from flat `src/` into a pnpm-workspace monorepo of scoped `@kuzo-mcp/*` packages without breaking functionality at any step.

### A.0 Scope

**In:** workspace layout, per-package `package.json`, tsconfig architecture, loader rewrite for dev/installed parity, dev workflow, risk matrix, parity test.
**Out:** provenance + publishing (Part B), install CLI (Part D). `publishConfig` fields land with Part A but are inert until Part B is wired.

### A.1 Step-by-step migration plan

Each step leaves the tree **buildable and runnable** (`pnpm typecheck && pnpm lint && pnpm build`, plus `pnpm start:mcp` boots and all three plugins load). Each step is one atomic commit on `phase-2.5e/supply-chain` (or a stacked sub-branch).

#### Step 1 — Tooling prereqs (no code moves)

- Add `packageManager: "pnpm@10.x"` to the root `package.json` (look up exact current version via `npm view pnpm version` — do not guess).
- Add `.npmrc` at repo root with `strict-peer-dependencies=true`, `auto-install-peers=false`, `link-workspace-packages=deep`, `prefer-workspace-packages=true`.
- Replace `package-lock.json` with `pnpm-lock.yaml` (`pnpm import` then `pnpm install`).
- **Verify green:** `pnpm install` succeeds; `pnpm build`, `pnpm typecheck`, `pnpm lint`, `pnpm start:mcp` still work. No source code has moved yet.

#### Step 2 — Introduce the workspace shell

- Create `pnpm-workspace.yaml` listing `packages/*` (dir does not exist yet — empty glob is fine).
- Create `tsconfig.base.json` at repo root. Existing `tsconfig.json` continues to build `src/` unchanged.
- **Verify green:** same as Step 1.

#### Step 3 — Extract `@kuzo-mcp/types` (leaf package, zero runtime deps)

- Create `packages/types/`.
- Move `src/plugins/types.ts` → `packages/types/src/index.ts`.
- Add `packages/types/package.json` + `packages/types/tsconfig.json` (both composite, type-only — see §A.2 and §A.3).
- In the root `package.json`, add a `workspace:*` dep on `@kuzo-mcp/types`.
- Rewrite all existing `src/**` imports of `../plugins/types.js` → `@kuzo-mcp/types`. Mechanical sed; use `tsc --noEmit` to find stragglers.
- Build order is now: `types` → root (`tsc -b` at root).
- **Verify green:** `pnpm -r build`, server boots, all 3 plugins load.

> **Why first:** `types` has no runtime code and no dependencies on any other workspace package. Extracting it first unblocks every later package without any circular-dep risk.

#### Step 4 — Extract `@kuzo-mcp/core`

- Create `packages/core/`.
- Move `src/core/**` → `packages/core/src/**`. That's: server, loader, plugin-process, plugin-host, ipc, consent, audit, credentials, registry, config, logger.
- Loader still references `"../plugins/<name>/index.js"` — **this will break**. Expected — the Step 5 loader rewrite fixes it.
- Loader rewrite (see §A.5): switch from path-based resolution to package-name resolution (`import("@kuzo-mcp/plugin-github")`).
- `plugin-host.js` path: switch to `import.meta.resolve("@kuzo-mcp/core/plugin-host")` + `fileURLToPath()` — see §A.5.
- **Verify green after both loader rewrite AND plugins have been extracted (Step 5).** Between 4 and 5 the tree intentionally won't build — treat 4+5 as a single commit if stacking is hard.

> **Uncertainty:** splitting 4/5 into two green commits is cleanest but requires a temporary compatibility shim in the loader that tries both the old path and new package name. If the shim is ugly, land 4+5 as one commit. **Recommendation: one commit.**

#### Step 5 — Extract the three plugin packages in parallel

Do each of: `plugin-github`, `plugin-jira`, `plugin-git-context`.

- `packages/plugin-<name>/src/**` = contents of `src/plugins/<name>/**`.
- Each package depends on `@kuzo-mcp/types` as a runtime import — but declared as **peerDependency** (see §A.7).
- Each plugin package gets a `kuzoPlugin` field in its `package.json` (§A.2).
- Update `kuzo.config.ts` if it references bare plugin names — plugin names stay the same (`"github"`, `"jira"`, `"git-context"`), the loader resolves them to scoped package names via a map (§A.5).
- **Verify green:** `pnpm -r build`, `pnpm start:mcp`, confirm `plugin.loaded` audit rows for all three.

#### Step 6 — Extract `@kuzo-mcp/cli`

- Create `packages/cli/`, move `src/cli/**` → `packages/cli/src/**`.
- `packages/cli/package.json` owns the `bin: { "kuzo": "./dist/index.js" }` entry.
- Root `package.json` drops its `bin` field.
- Shebang in `packages/cli/src/index.ts` stays `#!/usr/bin/env node`. Ensure build preserves file mode (`chmod +x dist/index.js` via a postbuild script, or use a shebang-preserving wrapper).
- **Verify green:** `pnpm --filter @kuzo-mcp/cli build && node packages/cli/dist/index.js --help`.

#### Step 7 — Delete the legacy `src/` tree

- After Steps 3-6 the old `src/` is empty. Remove it. Remove the root `tsconfig.json`'s `include`/`rootDir` for `src/`; root tsconfig becomes solution-style (`{ files: [], references: [...] }`).
- Root `package.json` keeps only workspace-level scripts and dev-only deps (ESLint, tsx for dev runs, TypeScript itself).
- **Verify green:** `pnpm install && pnpm -r build && pnpm -r test && pnpm start:mcp`.

#### Step 8 — Wire project references for incremental builds

- Each per-package `tsconfig.json` adds `references: [{ path: "../types" }]` etc.
- Root script: `"build": "tsc -b"`.
- **Verify green:** second `pnpm build` is a no-op (incremental); modifying a file in `types/` rebuilds only `types` + direct dependents.

#### Step 9 — CI + formatting

- Update any CI workflow to `pnpm install --frozen-lockfile` and `pnpm -r build`.
- Single root ESLint config — confirm it still covers `packages/**`.
- Add ESLint rule: no cross-plugin imports between `packages/plugin-*`.
- **Verify green:** CI passes.

#### Step 10 — Dev-to-install parity test (integration)

- Add a test script that: (a) `pnpm pack`s each `@kuzo-mcp/plugin-*`, (b) installs the tarball into a temp dir mimicking `~/.kuzo/plugins/<name>/`, (c) boots the server with `KUZO_PLUGINS_DIR=<tempdir>`, (d) asserts all three plugins load.
- This is the contract test for §A.8 parity.
- **Verify green:** the new integration script passes locally and in CI.

**Total: 10 steps.** Steps 4+5 may merge into one if the stacked compat shim is too ugly — implementer's call.

### A.2 Per-package `package.json` blueprints

Common fields across all packages: `"type": "module"`, `"engines.node": ">=20.0.0"`, `"license": "MIT"`, `"repository": { "type": "git", "url": "git+https://github.com/seantokuzo/seantokuzo-mcp.git", "directory": "packages/<name>" }`, `"publishConfig": { "access": "public", "provenance": true }`.

`publishConfig.provenance` is inert until Part B wires the release workflow. `repository.directory` is **load-bearing** for provenance — Sigstore matches OIDC claims against this field. Monorepo `directory` is mandatory.

**`kuzoPlugin` metadata shape:**

| Field | Purpose |
|-------|---------|
| `name` | Canonical plugin name as used in `kuzo.config.ts` (`"github"`) — decouples config keys from package names |
| `permissionModel` | `1` for V2, omitted for V1 (redundant with code but lets the loader pre-filter before importing) |
| `entry` | Relative path to the dist entry (lets us override main without breaking publish tooling) |
| `minCoreVersion` | Semver of `@kuzo-mcp/core` the plugin was built against — loader warns on mismatch |

`@kuzo-mcp/types` as a **peerDependency**: plugins don't need their own copy at runtime — core owns the contract. This avoids type-identity mismatches (two copies of `KuzoPlugin` interface that are structurally identical but nominally different). See §A.7.

#### Root `tsconfig.json` (solution-style, post-Step 7)

```json
{
  "files": [],
  "references": [
    { "path": "./packages/types" },
    { "path": "./packages/core" },
    { "path": "./packages/cli" },
    { "path": "./packages/plugin-github" },
    { "path": "./packages/plugin-jira" },
    { "path": "./packages/plugin-git-context" }
  ]
}
```

#### Key rules

- **`composite: true`** is required on every referenced project (enforced by TS 5.5). Forces `declaration: true`, requires explicit `include`, enables incremental build info.
- **Build command:** `tsc -b` at root, always. It walks the reference graph, only rebuilds out-of-date projects, respects dependency order.
- **`module: NodeNext` + `moduleResolution: NodeNext` + `exports` map:** TS resolves `import "@kuzo-mcp/core"` through the package's `exports.".".types` field (not `main` / `types`), which is why every package lists both `types` and `import` conditions in `exports`. Subpath imports (`"@kuzo-mcp/core/plugin-host"`) require explicit subpath entries in `exports` — there's no wildcard fallback under NodeNext.
- **Dev vs build resolution:** in dev, pnpm symlinks `node_modules/@kuzo-mcp/core` → `packages/core`. The package's `exports.import` points at `./dist/index.js`. So **you must have run `tsc -b` at least once** before `node` can resolve cross-package runtime imports. For iteration, run `tsc -b --watch` in one terminal and `node dist/server.js` in another.
- **tsx for pure dev:** optional follow-up. Add a `"development": "./src/index.ts"` condition to each `exports` block so `pnpm dev:cli` runs TS directly without pre-building. Not required for 2.5e.

### A.5 Loader rewrite (the crux)

#### Current behavior (flat repo)

```ts
const pluginPath = resolve(__dirname, "..", "plugins", name, "index.js");
await import(pathToFileURL(pluginPath).href);
```

Path-based. After the split, there is no `../plugins` — plugins are separate packages.

#### New behavior (package-name resolution with dual-mode fallback)

**Logical name → package name map** lives in `@kuzo-mcp/core` (not env-var driven; decoupling internal plugin names from npm package names is a **security property** — config can never be coerced into pointing at an arbitrary package).

```ts
// packages/core/src/plugin-resolver.ts (new file)
const BUILTIN_PLUGINS: Record<string, string> = {
  "github":      "@kuzo-mcp/plugin-github",
  "jira":        "@kuzo-mcp/plugin-jira",
  "git-context": "@kuzo-mcp/plugin-git-context",
};
```

Third-party plugins declare their package name explicitly in config:

```ts
// kuzo.config.ts
plugins: {
  "my-plugin": { enabled: true, packageName: "@someone-else/kuzo-plugin-foo" }
}
```

#### Resolution algorithm

```ts
async function resolvePluginEntry(name: string): Promise<string> {
  const pkg = BUILTIN_PLUGINS[name] ?? config.plugins[name]?.packageName;
  if (!pkg) throw new Error(`Unknown plugin: ${name}`);

  // 1. Try installed-mode: ~/.kuzo/plugins/<name>/node_modules/<pkg>/
  const installedRoot = process.env.KUZO_PLUGINS_DIR
    ?? join(homedir(), ".kuzo", "plugins");
  const installedPath = join(installedRoot, name, "node_modules", pkg);
  if (existsSync(join(installedPath, "package.json"))) {
    return pathToFileURL(
      join(installedPath, readMainEntry(installedPath))
    ).href;
  }

  // 2. Fall back to dev-mode (workspace symlink resolved via import.meta.resolve)
  //    Works because pnpm symlinks <repo>/node_modules/<pkg> → packages/<name>.
  return import.meta.resolve(pkg);
}
```

**Why `import.meta.resolve`, not `require.resolve`:** the whole server is ESM. `require.resolve` requires CJS interop. `import.meta.resolve` is stable in Node 20+ (synchronous since 20.6.0). Returns a `file://` URL string.

#### `plugin-host.js` path

Currently `resolve(__dirname, "plugin-host.js")`. After the split, `__dirname` is `.../node_modules/@kuzo-mcp/core/dist/`. Switch anyway:

```ts
import { fileURLToPath } from "url";

const hostPath = fileURLToPath(
  import.meta.resolve("@kuzo-mcp/core/plugin-host")
);
fork(hostPath, [], { ... });
```

Reads the path through the `exports` map (`"./plugin-host": "./dist/plugin-host.js"`):
- Reorganizing `dist/` only touches the `exports` map.
- Works identically whether core is linked (dev) or installed (prod).
- Self-documenting — the subpath is a public API surface.

`fork()` needs a filesystem path, not a URL. `fileURLToPath()` handles that.

#### V1/V2 gating

Unchanged at runtime. The `kuzoPlugin.permissionModel` field in `package.json` (§A.2) lets the loader refuse to import a V1 plugin under `KUZO_STRICT=true` *before* running plugin code. Recommended optimization.

### A.6 Dev workflow

```bash
pnpm install                # hoists + symlinks workspace packages
pnpm build                  # = tsc -b
pnpm build:watch            # = tsc -b --watch
pnpm start:mcp              # = pnpm --filter @kuzo-mcp/core exec node dist/server.js
pnpm dev:cli                # = pnpm --filter @kuzo-mcp/cli exec tsx src/index.ts
```

**Hot-reload a plugin during dev:**
- Edit → save → `tsc -b --watch` rebuilds.
- Running server does **NOT** pick up the change — `PluginProcess` has cached the forked child.
- Must restart the server (or kill the child; it'll auto-restart and pick up fresh bytes, but that's a race).
- Live HMR is explicitly out of scope (CLAUDE.md anti-patterns).

**Adding a new first-party plugin:**
1. `mkdir packages/plugin-foo`; copy the template from §A.2.
2. Add entry to `BUILTIN_PLUGINS` in `@kuzo-mcp/core`.
3. Add reference to root `tsconfig.json`.
4. `pnpm install`; enable in `kuzo.config.ts`.

### A.7 Risks + mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Hardcoded `__dirname` in loader + plugin-process | **High** | Switch to `import.meta.resolve` + `exports` map (§A.5). |
| ESM + `exports` + pnpm symlinks — phantom deps | Med | `strict-peer-dependencies=true` in `.npmrc` catches accidental hoisted-dep reliance. ESLint `no-restricted-imports` rule. |
| Circular deps (`core` ↔ `types` ↔ plugins) | Med | Enforce: `types` has zero workspace deps. `core` depends on `types`. Plugins depend on `types` (peer). Plugins MUST NOT depend on `core` or each other. Loader cross-plugin calls go through IPC. |
| Bin shebang + file mode after tsc build | Med | `packages/cli/package.json` postbuild: `"postbuild": "chmod +x dist/index.js"`. |
| Source maps point at wrong paths | Low | `declarationMap: true` + `sourceMap: true` in base config; paths are relative to each package root. |
| `import type` only — do plugins need `@kuzo-mcp/types` at runtime? | Med | `types` exports `defineTool`/`isV2Plugin` as runtime values. Plugins need it at runtime. Declared as `peerDependency`; core provides the singleton copy — avoids type-identity drift. |
| Tests with relative paths | Low | Run `rg "from ['\"]\.\." packages/**/src/**` after each move. |
| `.env` loading location | Med | `configManager` reads via `dotenv.config()`. Keep `.env` at repo root (dev) and at `process.cwd()` in installed mode. |
| pnpm hoisting surfacing a transitive dep we didn't declare | Med | `.npmrc`: `strict-peer-dependencies=true`. Default pnpm behavior is strict isolation. |
| `workspace:*` protocol in published package | **High** | pnpm rewrites `workspace:*` → concrete version at publish (`pnpm publish` does this automatically). **Verify by `pnpm pack` then inspecting the tarball's `package.json`** before ever running `pnpm publish`. |
| Plugin's `kuzoPlugin.entry` drifting from `main` | Low | Loader prefers `kuzoPlugin.entry` over `main` if both present. Single source of truth: `main`. |

### A.8 Dev-to-install parity

A plugin that works in dev (workspace-symlinked) must also work when published to npm and installed via `kuzo plugins install`. Contract test lands in Step 10:

```
scripts/test-install-parity.mjs
  1. pnpm --filter @kuzo-mcp/plugin-github pack
     → produces kuzo-mcp-plugin-github-0.0.1.tgz
  2. mkdir -p $TMPDIR/kuzo-install-test/github
```

**Why non-negotiable:** dual-mode resolution (§A.5) is the single most likely thing to break silently. This test exercises the installed-mode branch with real tarballs, catching issues the dev-mode (symlink) branch would hide — missing files from `files` whitelist, broken `exports` subpaths, peer-dep resolution failures, shebang/file-mode regressions.

Run on every PR touching `packages/*/package.json` or the loader.

### B.7 Dry-run + bootstrap strategy

**Canary order (lowest blast radius first):**

1. **`@kuzo-mcp/types`** — pure types, zero runtime, no external deps. If publish breaks, nothing is broken for real users. **Recommended canary.**
2. `@kuzo-mcp/plugin-git-context` — no external API deps, no auth, just filesystem. Low stakes.
3. `@kuzo-mcp/core` — depends on `types` only. Publishes next so plugins have a core to target.
4. `@kuzo-mcp/plugin-github`, `@kuzo-mcp/plugin-jira` — parallel.
5. `@kuzo-mcp/cli` — last. Consumes everything above.

**Before touching prod — snapshot releases on a branch:**

```bash
# Branch: phase-2.5e/release-bootstrap
pnpm changeset version --snapshot preview
pnpm release   # publishes @kuzo-mcp/types@0.0.0-preview-<timestamp>
```

- `--snapshot` mints a timestamped version (`0.0.0-preview-20260413...`).
- `changeset publish --tag preview` tags it as `preview`, not `latest`. Users running `pnpm add @kuzo-mcp/types` don't see it.
- Iterate until green, then back out to clean `main` and start real publishes.

**First real version: `0.0.1`, not `1.0.0`.** Signals pre-stable clearly. Changesets defaults to minor bumps from `0.x`, so breaking changes are `0.0.1 → 0.0.2` during this phase — intentional, keeps us below `1.0`.

**Rollback playbook:**

- Bad publish within 72h: `npm unpublish @kuzo-mcp/<name>@<version>`.
- After 72h: `npm deprecate @kuzo-mcp/<name>@<version> "broken — use X"` and publish a fixed patch.
- Never republish the same version after unpublishing — npm blocks this for 24h and breaks anyone who installed the bad version.

## Part C — Pre-install provenance verification

> The implementer's guide to Option A: verify npm provenance **before** any install script runs. Core supply-chain defense for `kuzo plugins install`.

**Target dependencies (verified 2026-04-13):** `sigstore@^4.1.0`, `@sigstore/verify@^3.1.0`, `@sigstore/bundle@^4.0.0`, `pacote@^21.5.0`. Node ≥ 20.17 or ≥ 22.9 (peer requirement of sigstore-js).

### C.1 Algorithm

User types `kuzo plugins install github`. The tool does:

```
1.  resolve name → "@kuzo-mcp/plugin-github" via BUILTIN_PLUGINS (Part A §A.5)
2.  read CLI flags (--version, --registry, --trust-unsigned, --offline, --allow-third-party, --allow-builder)
3.  dryResolve: pacote.manifest("@kuzo-mcp/plugin-github@<range>") WITHOUT install
       → get {_id, version, _integrity, dist.tarball, dist.attestations}
4.  if (!dist.attestations):
       if (--trust-unsigned) warn-loud-and-continue
       else exit(10) "no provenance attestation"
5.  fetchAttestations: GET <dist.attestations.url on current registry host>
       → { attestations: [{predicateType, bundle}, ...] }
6.  for each attestation:
       a. decodeSLSA: parse in-toto statement from bundle.dsseEnvelope.payload (base64)
       b. verifySubject: statement.subject[0].name === purl(pkg) AND sha512 === hex(_integrity)
       c. verifySigstore: sigstore.verify(bundle, { tufCachePath, tufForceCache, keySelector? })
       d. evaluatePolicy: apply TrustPolicy to SLSA provenance attestation
               → classify first-party vs third-party, block disallowed builders
7.  if any check fails: exit with numeric code + recoverable message (§C.7)
8.  downloadTarball: pacote extracts to ~/.kuzo/plugins/<name>/.tmp/pkg/ (no install scripts)
9.  recordVerification: ~/.kuzo/plugins/<name>/.tmp/verification.json (frozen evidence)
10. parseManifest: import pkg/dist/index.js, read KuzoPlugin manifest (capabilities, version)
11. consentFlow: invoke existing 2.5c consent code against the new plugin
12. registerPlugin: atomic rename .tmp/ → <version>/; add to kuzo.config.ts; update ~/.kuzo/plugins/index.json
13. exit(0) with summary: "Installed github@1.2.0 (first-party, 2 attestations verified)"
```

Every `exit(N)` with N ≠ 0 leaves `~/.kuzo/plugins/<name>/` **unchanged** (atomic install: stage to `.tmp/`, rename on success).

### C.2 npm registry attestation API

**Endpoint (verified against `pacote@21.5.0` source):** the packument's `dist.attestations.url` is authoritative. In practice:

```
https://registry.npmjs.org/-/npm/v1/attestations/<name>@<version>
```

Scoped packages use the URL-encoded form: `@scope%2fname@<version>` (percent-encoded `/`). `dist.attestations.url` handles this — **always use it**, do not reconstruct the URL. Pacote rewrites the host to the current configured registry so mirrors work: `new URL(dist.attestations.url).pathname` onto `this.registry`. Spec requires same rewrite.

**Response shape:**

```ts
type AttestationsResponse = {
  attestations: Array<{
    predicateType: string;   // e.g. "https://slsa.dev/provenance/v1"
                             //   or "https://github.com/npm/attestation/tree/main/specs/publish/v0.1"
    bundle: SigstoreBundle;  // Sigstore v0.1/0.2/0.3 bundle; contains dsseEnvelope + verificationMaterial
  }>;
};
```

**Expected attestations per publish:**
- Exactly one SLSA provenance (`slsa.dev/provenance/v1`) — keyless, Fulcio cert inside the bundle.
- Exactly one npm publish attestation (`npm/attestation/tree/main/specs/publish/v0.1`) — keyed, `bundle.dsseEnvelope.signatures[0].keyid` references a key from `/-/npm/v1/keys`.

Both MUST verify. Pacote enforces; we mirror.

**Error shapes:**
- `404` → no attestations published. Map to exit code 10.
- `5xx` / network error / timeout → recoverable, exit code 11 (suggest retry).
- Missing `attestations` field or empty array → treat as 404.
- Malformed bundle (JSON parse fail) → exit code 12 "corrupt attestation".

**Reference implementation:** `npm/cli` defers entirely to `pacote.manifest({verifyAttestations: true})` in `pacote/lib/registry.js` (`latest`). That file is gold standard — mirror its logic.

### C.3 `sigstore.verify()` usage

Pacote calls **`sigstore.verify(bundle, options)`** from the `sigstore` meta-package — NOT `@sigstore/verify` directly. `@sigstore/verify` is lower-level (`Verifier` class, policy objects); the `sigstore` wrapper is the stable consumer API.

```ts
import { verify } from 'sigstore';
import type { Bundle } from '@sigstore/bundle';

await verify(bundle, /* payload? */ undefined, {
  ctLogThreshold: 1,                // default: 1 CT log
  tlogThreshold: 1,                 // default: 1 Rekor inclusion
  certificateIssuer: 'https://token.actions.githubusercontent.com',
  certificateIdentityURI: 'https://github.com/seantokuzo/seantokuzo-mcp/.github/workflows/release.yml@refs/heads/main',
  tufCachePath: '~/.kuzo/tuf-cache',
  tufForceCache: true,              // offline-resilient; refresh via kuzo plugins refresh-trust-root
  keySelector: publicKey ? () => publicKey.pemkey : undefined,
});
```

Key points:
- `verify()` **throws** on any failure. Don't check a return value. Wrap in try/catch.
- For DSSE bundles (attestations), omit the `payload` argument — statement is inside the envelope.
- `certificateIssuer` + `certificateIdentityURI` perform **cryptographic identity pinning** against the Fulcio cert SAN. For first-party, pin the exact workflow URI. For third-party, omit these options and rely on SLSA policy evaluation in §C.5.
- `keySelector` used for **keyed** attestations (the publish attestation). Fetch keys from `<registry>/-/npm/v1/keys` or Sigstore TUF target `<host><pathname>/keys.json`.
- `tufForceCache: true` avoids network fetches of the Sigstore TUF root on every install. Ship a refresh subcommand (§C.8).

**Two orthogonal verification layers — do not conflate:**

| Layer | Purpose | Tool |
|-------|---------|------|
| (a) Sigstore signature verification | "Bundle's signature is cryptographically valid and logged to Rekor." | `sigstore.verify()` |
| (b) SLSA provenance policy | "This verified bundle claims the package was built by a trusted workflow in a trusted repo." | Our `TrustPolicy` (§C.5) over decoded in-toto statement |

(a) must pass before (b) runs. (b) is policy over already-verified claims.

### C.4 SLSA payload parsing

Decode the in-toto statement **before** calling `sigstore.verify` so we can match subject/integrity up front (mirrors pacote):

```ts
const statement = JSON.parse(
  Buffer.from(bundle.dsseEnvelope.payload, 'base64').toString('utf8')
) as InTotoStatement;
```

Shape (SLSA v1.0 / in-toto v1):

```ts
type InTotoStatement = {
  _type: 'https://in-toto.io/Statement/v1';
  subject: Array<{ name: string; digest: { sha512: string } }>;
  predicateType: 'https://slsa.dev/provenance/v1';
  predicate: {
    buildDefinition: {
      buildType: string;                // e.g. 'https://slsa-framework.github.io/github-actions-buildtypes/workflow/v1'
      externalParameters: {
        workflow: {
          ref: string;                  // 'refs/heads/main' | 'refs/tags/v1.2.0'
          repository: string;           // 'https://github.com/seantokuzo/seantokuzo-mcp'
          path: string;                 // '.github/workflows/release.yml'
        };
      };
      internalParameters?: unknown;
      resolvedDependencies?: Array<{ uri: string; digest: Record<string, string> }>;
    };
    runDetails: {
      builder: { id: string };          // 'https://github.com/actions/runner/github-hosted'
      metadata: { invocationId: string; startedOn: string; finishedOn: string };
    };
  };
};
```

Subject check is non-negotiable:
1. `statement.subject[0].name` must equal the package PURL: `pkg:npm/<name>@<version>` (URL-escape scope).
2. `statement.subject[0].digest.sha512` (hex) must equal `ssri.parse(dist.integrity).hexDigest()`.

Repo-origin check uses `externalParameters.workflow.repository` parsed via `new URL()`. Comparing host + first path segment gives org; second gives repo.

### C.5 Trust policy implementation

```ts
// packages/core/src/provenance/policy.ts
export type TrustPolicy = {
  allowedBuilders: string[];   // exact-match prefixes; e.g. ['https://github.com/actions/runner']
  firstPartyOrgs: string[];    // GitHub orgs that mark a plugin "first-party"; e.g. ['seantokuzo']
  requireProvenance: boolean;  // default true; false only via --trust-unsigned
  allowThirdParty: boolean;    // default true (v1); policy-gated in future
};

export type PolicyResult =
  | { verified: true; firstParty: boolean; repo: string; builder: string }
  | { verified: false; code: string; reason: string };

export function evaluate(
  statement: InTotoStatement,
  policy: TrustPolicy,
): PolicyResult {
  if (statement.predicateType !== 'https://slsa.dev/provenance/v1') {
    return { verified: false, code: 'E_WRONG_PREDICATE',
             reason: `expected SLSA provenance, got ${statement.predicateType}` };
  }

  const builder = statement.predicate.runDetails?.builder?.id ?? '';
  if (!policy.allowedBuilders.some(b => builder === b || builder.startsWith(b + '/'))) {
    return { verified: false, code: 'E_DISALLOWED_BUILDER',
             reason: `builder '${builder}' not in allow-list` };
  }

  const repoURL = statement.predicate.buildDefinition?.externalParameters?.workflow?.repository;
  if (!repoURL) {
    return { verified: false, code: 'E_MISSING_REPO', reason: 'workflow.repository absent' };
  }

  let parsed: URL;
  try { parsed = new URL(repoURL); }
  catch { return { verified: false, code: 'E_MALFORMED_REPO', reason: `invalid URL: ${repoURL}` }; }

  if (parsed.host !== 'github.com') {
    return { verified: false, code: 'E_UNSUPPORTED_HOST',
             reason: `only github.com supported, got ${parsed.host}` };
  }

  const org = parsed.pathname.split('/').filter(Boolean)[0] ?? '';
  const firstParty = policy.firstPartyOrgs.includes(org);

  if (!firstParty && !policy.allowThirdParty) {
    return { verified: false, code: 'E_THIRD_PARTY_BLOCKED',
             reason: `third-party plugins disabled; ${org} not in firstPartyOrgs` };
  }

  return { verified: true, firstParty, repo: repoURL, builder };
}
```

Default policy constant for Kuzo:

```ts
export const DEFAULT_POLICY: TrustPolicy = {
  allowedBuilders: ['https://github.com/actions/runner'],
  firstPartyOrgs: ['seantokuzo'],
  requireProvenance: true,
  allowThirdParty: true,
};
```

### C.6 Install flow integration

**Never run install scripts before verification.** Sequence:

1. **Resolve-only** (`pacote.manifest`) — reads packument; no tarball, no scripts. Returns `_integrity`, `dist.tarball`, `dist.attestations.url`.
2. **Fetch attestations** (§C.2). Parse + verify (§C.3–C.5) entirely **in memory**.
3. **Record verification** to `~/.kuzo/plugins/<name>/.tmp/verification.json` (§C.8 schema).
4. **Download tarball** via `pacote.extract(spec, target, {...opts, Arborist: null})`. Pacote extract itself does not run scripts — scripts happen when `npm install` runs lifecycle. For our layout: extract to `.tmp/pkg/` only.
5. **Parse manifest** — dynamic-import `pkg/dist/index.js`, validate `KuzoPlugin` shape (Zod schema on `capabilities`, `permissionModel`, `name`, `version`). Only AFTER verification.
6. **Install transitive deps:** `npm install --prefix=<stagingDir> --ignore-scripts --no-audit --no-fund` into `.tmp/node_modules/`.
7. **Consent** — delegate to existing 2.5c consent flow with plugin manifest.
8. **Atomic commit** — `fs.rename(.tmp, <version>)`; update `~/.kuzo/plugins/<name>/current` symlink → `<version>`; update `~/.kuzo/plugins/index.json` registry.

**Directory layout after success:**

```
~/.kuzo/plugins/
├── index.json                  # { "github": { version, integrity, source, installedAt } }
├── .lock                       # install/update concurrency lock (Part D §D.6)
├── tuf-cache/                  # shared sigstore TUF cache
├── attestations-cache/         # §C.8
└── github/
    ├── current → 1.2.0/        # symlink to active version
    ├── 1.2.0/
    │   ├── verification.json   # frozen evidence
    │   ├── pkg/                # extracted tarball root
    │   │   ├── package.json
    │   │   └── dist/index.js
    │   └── node_modules/       # transitive deps (installed with --ignore-scripts)
    └── 1.1.8/                  # retained prior version for rollback (keep last 3)
        └── ...
```

### C.7 Failure modes

| Scenario | Exit | CLI message | Recoverable? | User fix |
|---|---|---|---|---|
| No attestation on package | `10` | `No provenance attestation for <pkg>@<ver>. Kuzo requires signed plugins.` | Yes | Use `--trust-unsigned` (loud warning) or ask publisher to re-publish with provenance |
| Registry network fail / timeout | `11` | `Could not reach npm registry to fetch attestations. Retry with network.` | Yes | Retry; check `--registry`, proxy, offline flag |
| Registry 5xx | `11` | `npm registry returned <code>. Attestation lookup failed.` | Yes | Retry; check registry status |
| Malformed attestation JSON | `12` | `Attestation payload is malformed (JSON parse failed).` | No | File issue with publisher; blocked |
| Invalid Sigstore signature | `13` | `Sigstore signature verification failed: <msg>. Package may be tampered.` | No | Do not override; report to npm security |
| Rekor inclusion proof fail | `13` | `Transparency log inclusion check failed for <pkg>.` | No | Same |
| TUF root fetch failure (first run) | `14` | `Could not fetch Sigstore trust root. Check network or run: kuzo plugins refresh-trust-root` | Yes | Retry or pre-seed TUF cache |
| Subject mismatch (PURL) | `15` | `Attestation signed a different package: expected <purl>, got <signed>.` | No | Blocked; likely registry tampering |
| Integrity mismatch (sha512) | `15` | `Tarball integrity doesn't match attestation subject.` | No | Blocked |
| Wrong builder (policy) | `16` | `Plugin built by <builder>, not in allowed-builders list.` | Yes (third-party) | `kuzo plugins install --allow-builder <url>` (requires consent) |
| Wrong repo (third-party blocked) | `17` | `Third-party plugin from <org> is blocked by current policy.` | Yes | `--allow-third-party` or add to `firstPartyOrgs` |
| Malformed SLSA payload | `18` | `SLSA predicate is missing required fields: <fields>.` | No | Blocked |
| Multiple attestations — one fails | `13-18` | (whichever failed) | Depends | All-or-nothing: every attestation must verify |
| `--trust-unsigned` used | `0` (warning) | `WARNING: installing <pkg> without provenance verification. This is a supply-chain risk.` | N/A | N/A |

Exit codes `10–19` reserved for provenance domain; documented in `packages/core/src/provenance/errors.ts`.

### C.8 Caching

**Cache only positive verifications**, keyed by `{name}@{version}@{integrity}`. Location: `~/.kuzo/attestations-cache/<sha256(key)>.json`.

```ts
type CachedVerification = {
  schemaVersion: 1;
  package: { name: string; version: string; integrity: string };
  verifiedAt: string;             // ISO 8601
  firstParty: boolean;
  repo: string;
  builder: string;
  policySnapshot: TrustPolicy;    // the policy in effect at verification time
  predicateTypes: string[];
};
```

**TTL: none.** An installed version is immutable by integrity. Invalidate only on:
- Policy change (snapshot mismatch → re-verify on next install/update).
- Trust-root rotation (`kuzo plugins refresh-trust-root` wipes this cache).
- Package uninstall (cache entry stays; pure evidence, not authority).

**Never cache the raw bundles or attestations payload.** Always re-fetch on install — attestations can grow (registry adds a new publish attestation). Cache is a post-verification receipt used during `kuzo plugins list --verify-cached`.

**Security implications:** treat the cache as non-authoritative. Attacker with write access to `~/.kuzo` can forge entries but still can't cause install because verification always re-runs on install paths. Cache only accelerates `kuzo plugins verify` read-only commands; those MUST re-verify when the cache is missing or the policy snapshot differs.

**TUF cache** (`~/.kuzo/tuf-cache`) separate, managed by `@sigstore/tuf`. Force-cache (`tufForceCache: true`) so installs don't depend on Sigstore being reachable. Refresh every 7 days via `kuzo plugins refresh-trust-root` or on user demand.

### C.9 Edge cases

- **Scoped packages:** same endpoint; npm percent-encodes the slash. Always consume `dist.attestations.url`.
- **Two attestations per publish:** one SLSA provenance (keyless, Fulcio) and one npm publish attestation (keyed, registry key). **Verify both.** Skipping the publish attestation means trusting the build but not npm's claim that this tarball is what was uploaded.
- **Dist-tags and pre-releases:** resolve tag → version via `pacote.manifest("pkg@next")`, then pin. `kuzo.config.ts` stores `version` (not tag). `kuzo plugins install pkg@beta` is allowed; tag floating is not.
- **Deprecated / yanked versions:** `pacote.manifest` surfaces `deprecated` field. Warn but allow install if user passes `--allow-deprecated`. Yanked versions return 404 on packument — normal exit-code-10 path.
- **Mirror registries:** respect `--registry` and npmrc. Rewrite `dist.attestations.url` host to the active registry; otherwise a proxy-only setup breaks.
- **Non-npm registry (GitHub Packages, Verdaccio):** out of scope for v1. Only npmjs.org currently produces SLSA provenance at scale. Gate with `if (registry !== 'https://registry.npmjs.org/') refuse unless --allow-registry <url>`.

### C.10 Known gotchas

1. **`sigstore.verify()` does not enforce subject or PURL** — our job. Calling `verify()` without first checking `statement.subject[0]` against the package means you verified a valid signature over a *different* artifact. Pacote checks subject **before** calling verify; mirror that ordering.
2. **Fulcio cert identity is a URL, not a repo URL.** Workflow ref lives in the cert's SAN as `https://github.com/<org>/<repo>/.github/workflows/<file>.yml@refs/heads/main`. Use `certificateIdentityURI` to pin; do NOT confuse with `externalParameters.workflow.repository`.
3. **Publish attestation requires registry keys.** DSSE signed with npm registry key (`bundle.dsseEnvelope.signatures[0].keyid`). Fetch keys from `/-/npm/v1/keys` and pass via `keySelector`. Skip this → `sigstore.verify` throws "no public key found." `@sigstore/tuf` also serves these at target path — prefer TUF when available, fall back to REST.
4. **`tufForceCache: true` is not a one-time thing.** First run MUST populate the cache. Ship a `kuzo plugins refresh-trust-root` that forces fresh fetch; run automatically on first install and every 7 days.
5. **SLSA predicateType differences:** npm may emit both `slsa.dev/provenance/v1` and older `v0.2`. Accept v1; reject v0.2 (different field layout — `invocation.configSource.uri` vs `externalParameters.workflow.repository`). Fail closed.
6. **Bundle version drift:** Sigstore bundles v0.1 / v0.2 / v0.3 have different schemas. `sigstore@4` handles all three transparently. Pin `sigstore` to `^4.1.0` minimum.
7. **`--ignore-scripts` is not enough if you already called `npm install` elsewhere.** Our flow must never call `npm install` or `pacote.extract` on the plugin before §C.6 step 4. Any prefetch utility that runs `install` scripts defeats the entire point.
8. **Attestation URL host rewriting:** if a user has a private mirror, `dist.attestations.url` may point at npmjs.org but the mirror may or may not proxy attestations. Rewrite to active registry host (pacote does this in `registry.js`). Fall back to npmjs.org **only** if user opts in via `--attestation-registry`.
9. **`sigstore.verify` returns on success; it throws on failure.** Return type is `Signer`, not boolean.
10. **Do NOT use `@sigstore/verify` directly in v1.** Lower-level API changes more often. Use `sigstore.verify()` — same one pacote uses. Re-evaluate in v2 if we need fine-grained policy.

## Part D — Plugin install CLI

> User-facing commands. Ties Parts A-C together into a working `kuzo plugins` subcommand tree.

### D.1 Command surface

All commands live under `kuzo plugins <subcommand>`, owned by `@kuzo-mcp/cli`. Commander tree:

```
kuzo plugins install <name>[@version]   # install and register
  --registry <url>                      # override npm registry (npmjs.org only in v1)
  --trust-unsigned                      # bypass provenance check (loud warning)
  --allow-third-party                   # override policy block for non-firstPartyOrg plugins
  --allow-builder <url>                 # add to allowedBuilders for this install
  --allow-deprecated                    # install a deprecated version anyway
  --dry-run                             # verify + print plan, don't write

kuzo plugins uninstall <name>           # remove plugin entirely (all retained versions)
  --keep-versions                       # keep retained versions on disk for later re-register

kuzo plugins update [<name>]            # update one plugin (or all if omitted)
  --to <version>                        # pin to specific version (default: latest)
  --dry-run                             # show what would change; no writes

kuzo plugins rollback <name> [<version>]  # restore previous or specified retained version
                                          # default: n-1 (the version before current)

kuzo plugins list                         # table of installed plugins, versions, status
  --verify                                # re-verify each against cached attestations
  --json                                  # machine-readable output

kuzo plugins refresh-trust-root           # force-refresh Sigstore TUF cache; clear attestation cache

kuzo plugins verify <name>                # re-run Part C verification against installed version
                                          # exits non-zero on policy/signature regression
```

**MVP subset:** `install`, `uninstall`, `update`, `rollback`, `list`, `refresh-trust-root`. `verify` lands in the same phase — it's mostly Part C code reused.

**Deferred (out of 2.5e):** `search`, `info`, `disable`/`enable` (toggle without uninstalling). Config-level enable/disable already exists in `kuzo.config.ts`.

### D.2 Install flow (glue for Parts A + C)

The algorithm in Part C §C.1 is the full install flow. What D owns:

- **Commander subcommand handler** in `packages/cli/src/commands/plugins-install.ts`.
- **User prompts** (inquirer) surfaced at the right points:
  - After policy evaluation, before any disk writes: summary card with `name`, `version`, `source repo`, `builder`, `first-party? yes/no`, `declared capabilities` (parsed from manifest). User confirms with `y/N`.
  - Consent prompts handled by existing 2.5c code — integrated at step 11.
- **Progress UX:** nanospinner for each stage: "Resolving → Verifying signature → Checking policy → Downloading → Extracting → Parsing manifest → Consent → Registering". Failure at any stage surfaces the exit-code message from §C.7.
- **Config mutation** (see §D.5).
- **Locking** (see §D.6).

### D.3 Update flow

`kuzo plugins update` with no args updates all installed plugins. With `<name>`, single plugin.

```
1. read ~/.kuzo/plugins/index.json → list of installed plugins + current versions
2. for each target plugin:
   a. pacote.manifest(pkg@latest) → fetch latest version and attestations
   b. if latestVersion === currentVersion: skip
   c. run full Part C verification (§C.1 steps 4-7) on the new version
   d. diff the new manifest's `capabilities` against the currently-consented set
       - if same or subset: re-use existing consent, proceed
       - if added capabilities: surface diff table, require re-consent (y/N)
       - if user declines: abort this plugin, continue with others
   e. atomic install of new version to <name>/<new-version>/
   f. flip <name>/current symlink to <new-version>/
   g. prune retained versions: keep last 3, delete older
   h. update index.json with new version + installedAt
3. summary: { updated: [...], skipped: [...], failed: [...] }
```

**Capability diff example:**

```
[github] update 1.2.0 → 1.3.0

Changes to declared capabilities:
  + network.github.com (new)
  ~ credentials.GITHUB_TOKEN (scope changed: repo → repo, workflow)
  - filesystem.write (removed)

This update requires your consent. Continue? [y/N]
```

**Never auto-update.** The CLI does not schedule, daemonize, or background-run updates. User must type `kuzo plugins update` manually. Kuzo will print a one-line nudge at server start if any plugin is outdated (`kuzo plugins list --check-outdated` on start), but never act.

### D.4 Rollback semantics

Retention: keep last 3 versions per plugin in `~/.kuzo/plugins/<name>/<version>/`.

```
kuzo plugins rollback github           # rollback to n-1 (the version before current)
kuzo plugins rollback github 1.1.8     # rollback to specific retained version
```

Algorithm:

```
1. validate: target version exists under <name>/ OR target is "previous" and n-1 is retained
2. re-run consent flow against target version's manifest (capabilities may differ from current)
   - rationale: rollback is NOT implicitly less risky than upgrade; the target version may have
     different capabilities from what the user consented to at install time
3. flip <name>/current symlink → <target-version>/
4. update ~/.kuzo/plugins/index.json: currentVersion = target
5. audit log: "plugin.rolled_back" with from/to versions
6. restart prompt: "You must restart the MCP server for this rollback to take effect."
```

**If n-1 is not retained (user pruned via `--keep-versions=0` or manual rm):** exit code 20 "no retained version to roll back to. Install explicitly: `kuzo plugins install github@<version>`." This is a valid recovery path — install a specific older version from npm and it rolls back transparently (as long as that version still exists on npm and verifies).

### D.5 Config mutation

`kuzo.config.ts` enables plugins:

```ts
export default {
  plugins: {
    github: { enabled: true },
    jira:   { enabled: true },
    "my-plugin": { enabled: true, packageName: "@someone-else/kuzo-plugin-foo" },
  }
};
```

`kuzo plugins install` must update this file without destroying user formatting or adjacent code. Options:

1. **TypeScript AST manipulation** (ts-morph) — preserves formatting exactly. Heavyweight dep.
2. **Read the file, parse JSON-like fragment, re-serialize** — destroys comments/formatting.
3. **User edits config manually; CLI only verifies + installs to `~/.kuzo/plugins/`** — simplest. After install, CLI prints: `Add { enabled: true } under plugins.github in kuzo.config.ts, then restart the server.`

**Recommendation: option 3 for MVP.** The manifest lives in `~/.kuzo/plugins/`; the config is explicit user opt-in. Prompt-based insert with ts-morph is a follow-up.

For **uninstall**, mirror the same rule: CLI removes files, prints reminder to remove from config.

Alternative: introduce `~/.kuzo/plugins/enabled.json` as a secondary enablement source the loader consults:
- `kuzo.config.ts` wins if it sets `enabled: true|false` explicitly
- `enabled.json` (CLI-managed) fills the gap for plugins not mentioned in config
- Config = source of truth for committed/shared setups; `enabled.json` = source of truth for ad-hoc installs

**Recommendation for MVP:** just edit `kuzo.config.ts` with option 3 (prompt the user). Revisit if users complain.

### D.6 Concurrency + locking

Concurrent `kuzo plugins install` / `update` invocations must not clobber each other.

- **Lock file:** `~/.kuzo/plugins/.lock` (exclusive, O_CREAT | O_EXCL).
- **Contents:** `{ pid: number, command: string, startedAt: ISO }`.
- **Acquire:** at the start of any write-command (install/uninstall/update/rollback/refresh-trust-root). Fail fast with exit code 30 if locked: `Another kuzo plugins operation is running (pid <n>, started <when>). Wait or kill that process.`
- **Stale detection:** if pid's process no longer exists (`process.kill(pid, 0)` throws ESRCH), treat lock as stale, acquire.
- **Release:** on process exit (including unclean): use `onExit` from `signal-exit` or equivalent to `unlink` the lock. Wrap the critical section in try/finally.

Read commands (`list`, `verify`) do not lock.

### D.7 State files

Single source of truth for "what's installed":

**`~/.kuzo/plugins/index.json`**

```ts
type PluginsIndex = {
  schemaVersion: 1;
  plugins: Record<string, {
    currentVersion: string;
    packageName: string;        // npm package name
    installedAt: string;        // ISO 8601
    lastUpdatedAt: string;
    source: "first-party" | "third-party";
    retainedVersions: string[]; // list of retained version dirs, sorted newest-first
    integrity: string;          // current version's integrity hash
  }>;
};
```

**`~/.kuzo/plugins/<name>/<version>/verification.json`** — defined in Part C §C.8 `CachedVerification`. Frozen evidence at install time.

**Schema migration:** bump `schemaVersion` on breaking changes; loader refuses to read unknown schema version and prints a `kuzo plugins migrate-state` hint (command not in scope for 2.5e; ship `schemaVersion: 1` and don't break it).

### D.8 Error UX principles

- Every exit code has a short CLI message AND a longer explanation available via `kuzo help errors <code>` (or a URL link to `docs/ERRORS.md`). Don't make the user google cryptic Sigstore errors.
- Recoverable errors suggest the exact command to run next.
- Irrecoverable errors name the specific reason (`Sigstore signature invalid` vs `provenance verification failed`) and point at reporting venues (npm security, package author) rather than suggesting overrides for non-overrideable cases.
- `--trust-unsigned` always triggers a multi-line warning block surrounded by boxen red border. Not silenceable.

### D.9 Part D open questions

- **Config mutation strategy** — MVP is "prompt user to edit config manually." Is that acceptable DX, or do we bite the bullet and bring in ts-morph for auto-edit? **Recommend MVP as-is; revisit if users complain.**
- **`--registry` in v1** — Currently gated to npmjs.org only. Is there a near-term need for private registry support (e.g. enterprise Verdaccio)? **Recommend defer; flag exists but blocks with explicit `--allow-registry <url>`.**
- **Version retention policy** — keep last 3. Configurable via `~/.kuzo/config.json`? **Recommend hardcoded for MVP; make configurable post-phase if needed.**
- **First-run UX** — after Part A+B ship, existing dev installs of Kuzo won't have `~/.kuzo/plugins/`. First invocation of `kuzo plugins install` should bootstrap the directory tree with correct permissions. Document the first-install message.
